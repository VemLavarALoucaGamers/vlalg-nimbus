# NbFileUpload (queue de upload)

Fila de uploads HTTP feita com `XMLHttpRequest` + `FormData`.

Este componente **não** abre seletor de ficheiros: ele recebe uma lista de `File[]` via prop `files` (por exemplo a partir do `@changed` do `NbInputFile`).

## Como usar (demo)

No `packages/vue-components/nb-inputs/src/App.vue`:

```vue
<NbInputFile
  :key="uploadDemoPickerKey"
  nb-id="file-upload-demo-picker"
  input-name="file-upload-demo-picker"
  :multiple="true"
  @changed="setUploadDemoFiles"
/ >

<NbFileUpload
  nb-id="file-upload-demo-queue"
  :files="uploadDemoFiles"
  :upload-url="'/__nb-upload/upload'"
  :concurrent="2"
  :max-retries="1"
  :min-progress-display-ms="1200"
  after-upload-mode="default"
  :reupload-on-start="true"
  :show-cancel-all="true"
/>
```

## Props principais

- `nbId` (String, required): id raiz.
- `files` (File[], default `[]`): lista de ficheiros para enviar.
- `uploadUrl` (String, default `''`): endpoint `POST`.
- `concurrent` (Number, default `2`): número de uploads em paralelo.
- `maxRetries` (Number, default `2`): retries por ficheiro quando falha.
- `retryDelayMs` (Number, default `1000`): delay entre tentativas.
- `formFieldName` (String, default `'file'`): nome do campo no `FormData`.
- `uploadMethod` (String, default `'POST'`).
- `requestHeaders` (Object): headers extra para o XHR.
- `autoStart` (Boolean, default `false`): se `true`, inicia ao aparecer `pending`.
- `minProgressDisplayMs` (Number, default `0`):
  - Ajuda a “animar” a barra quando o browser envia ficheiros muito pequenos e o `onprogress` pode ser quase instantâneo (0→100).
- `afterUploadMode` (String, default `'default'`):
  - `default`: não limpa no fim.
  - `clear-on-success`: limpa **só** se todos os ficheiros terminarem em `success`.
  - `lock-until-clear-all`: quando tudo terminar e houve pelo menos um `success`, bloqueia o botão Start até limpar/remover.
- `reuploadOnStart` (Boolean, default `true`):
  - Ao clicar em Start, se for `true`, os itens finalizados (`success`, `error`, `cancelled`) voltam para `pending` para re-enviar.
  - Importante: em `lock-until-clear-all`, o Start pode ficar desabilitado até `clearAll`/lista vazia.

## Eventos

- `upload-progress`: `{ file, progress, index }`
- `upload-complete`: `{ file, response, index }`
- `upload-error`: `{ file, error, index }`
- `upload-all-complete`: `{ items: [{file,state,index}], allSuccess, allSettled }`
- `update:files`: emitido ao executar ações internas como `clearAll` ou remover item.

## Métodos expostos (ref)

- `startUpload()`
- `cancelRow(row)` (cancelar um XHR específico)
- `cancelAllActive()` (cancelar só os que estão em `uploading`)
- `cancelAllItems()` (cancelar `uploading` e marcar `pending` como `cancelled`)
- `clearAll()`
- `removeQueueItem(row)`
- `retryRow(row)`
- `retryAllErrors()`

## Nota sobre cancelamento vs apagar ficheiro no servidor

No teste local, o servidor tenta apagar o ficheiro quando detecta abort do cliente (`DELETE_ON_CLIENT_ABORT=true`).
Em uploads reais, se o cancel acontecer tarde (depois do servidor já ter completado), pode ser impossível “desfazer” o upload.
